# Contributing to FlowPilot

First off, thank you for considering contributing to FlowPilot! It's people like you that make open-source software such a great community.

## 1. Where do I go from here?

If you've noticed a bug or have a feature request, make one! It's generally best if you get confirmation of your bug or approval for your feature request this way before starting to code.

## 2. Fork & create a branch

If this is something you think you can fix, then fork FlowPilot and create a branch with a descriptive name.

### Branch Naming Conventions
* `feature/your-feature-name` (For new features)
* `fix/issue-description` (For bug fixes)
* `docs/update-readme` (For documentation changes)
* `refactor/clean-auth-flow` (For refactoring code)

## 3. Commit your changes

Make your changes and commit them with a descriptive commit message. We follow the Conventional Commits standard.

### Commit Message Format
```
type(scope): subject
```
* **feat**: A new feature
* **fix**: A bug fix
* **docs**: Documentation only changes
* **style**: Changes that do not affect the meaning of the code (white-space, formatting, etc)
* **refactor**: A code change that neither fixes a bug nor adds a feature
* **perf**: A code change that improves performance

## 4. Submitting a Pull Request

* Ensure your code passes all linting (`npm run lint` in client).
* Keep your PRs focused and as small as possible.
* Use the provided PR template.
* Include screenshots if your PR introduces visual changes.

## 5. Issue Reporting

When reporting an issue, please include:
* Your OS and Browser version.
* The exact steps to reproduce the issue.
* Expected behavior vs Actual behavior.
* Console logs or screenshots if applicable.
